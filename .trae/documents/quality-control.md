# AI Album 质量控制规范

## 1. 质量控制体系总览

### 1.1 质量目标

| 维度 | 目标 | 量化指标 |
|------|------|----------|
| 功能质量 | 产品功能符合PRD定义 | P0功能100%通过, P1功能≥95%通过 |
| 性能质量 | 满足PRD性能指标 | 首页≤2s, 搜索≤1s(文本)/≤3s(语义), API≤500ms |
| 安全质量 | 无高危及以上安全漏洞 | OWASP Top 10零漏过 |
| 可用性质量 | 服务稳定运行 | ≥99%可用性, 故障恢复≤5分钟 |
| 代码质量 | 代码规范一致 | 后端覆盖率≥80%, 前端≥60%, ML≥50% |
| 兼容性质量 | 多平台多浏览器可用 | Chrome/Firefox/Safari/Edge 90+, Ubuntu 22.04 ARM64 |
| 交付质量 | 每个迭代可发布 | 零P0遗留, P1遗留≤2个 |

### 1.2 质量控制原则

1. **左移原则**: 缺陷发现越早修复成本越低, 在设计/编码阶段即嵌入质量检查
2. **自动化原则**: 能自动化的检查不依赖人工, CI/CD流水线强制执行
3. **门禁原则**: 每个阶段设置质量门禁, 不达标不进入下一阶段
4. **可追溯原则**: 每个质量检查点有明确标准、执行方式、通过条件和记录
5. **持续改进原则**: 每次迭代回顾质量数据, 优化检查点和标准

| 规范项 | 标准 | 检查方式 | 强制级别 |

| 接口命名 | PascalCase, 无I前缀 (如`Photo`而非`IPhoto`) | 代码审查 | 强制 |

#### 10.1.3 Nginx金丝雀配置

```nginx
# /etc/nginx/conf.d/aialbum.conf
upstream aialbum_api {
    # 金丝雀发布: 通过cookie或header控制流量分配
    # 默认路由到v1(旧版本)
    server aialbum-server-v1:3000 weight=95;
    server aialbum-server-v2:3000 weight=5;
}

server {
    listen 80;
    server_name _;

    # 金丝雀用户识别(通过cookie)
    # canary=true的用户始终路由到v2
    set $backend "aialbum_api";
    if ($cookie_canary = "true") {
        set $backend "aialbum_server-v2:3000";
    }

    location / {
        proxy_pass http://$backend;
        proxy_set_header Host $host;
        proxy_set_header X-Real-IP $remote_addr;
        proxy_set_header X-Canary $cookie_canary;
    }

    # Admin可手动设置金丝雀cookie
    location /api/v1/admin/canary {
        internal;
        add_header Set-Cookie "canary=true; Path=/; Max-Age=86400";
    }
}
```

#### 10.1.5 自动化回滚规则

| 触发条件 | 回滚动作 | 通知 |

#### 10.1.6 数据库迁移兼容性要求

金丝雀发布期间v1和v2同时运行，数据库迁移必须向后兼容：

| 迁移类型 | 是否兼容 | 说明 |
|----------|----------|------|
| 新增表 | ✅ 兼容 | v1不读取新表 |
| 新增列(可空或有默认值) | ✅ 兼容 | v1忽略新列 |
| 新增列(NOT NULL无默认值) | ❌ 不兼容 | v1插入会失败，必须分两步迁移 |
| 重命名列 | ❌ 不兼容 | 使用新增列+数据迁移+旧列保留策略 |
| 删除列 | ❌ 不兼容 | 金丝雀期间保留，全量后下个版本删除 |
| 修改列类型 | ❌ 不兼容 | 使用新增列+应用层双写 |

**不兼容迁移的分步策略:**
1. 版本N: 新增列(可空)，应用层双写新旧列
2. 版本N+1: 数据迁移(旧列→新列)，读取新列
3. 版本N+2: 删除旧列(全量部署后)

| 环境差异 | 开发/生产环境配置不同 | 统一配置管理, Staging环境验证 |
| 性能问题 | 大数据量下性能劣化 | 性能基准测试, 大数据集测试 |

### 13.3 质量检查点演进

每个迭代结束时评估:
1. 现有检查点是否有漏过缺陷 → 补充检查点
2. 现有标准是否过于严格/宽松 → 调整阈值
3. 是否有新的质量风险 → 新增检查点
4. 自动化程度是否足够 → 增加自动化检查
